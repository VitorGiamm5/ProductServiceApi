# Architecture Decisions

## Redis

- Keep one Redis instance.
- Keep DB0 only.
- Use key prefixes for feature granularity instead of separate Redis databases.
- Use ACL users for read, write, metrics, and admin access.
- Keep Redis externally bound to localhost only.
- Use persistence, maxmemory, metrics, and circuit breaker.

## Cache

- Cache should be optional and resilient.
- Redis failures must not break API responses.
- Warmup is feature-based and configurable.
- Cache metrics should be exposed for Prometheus/Grafana.

## Idempotency

- Future features that add writes, retries, Polly policies, background publication, or message consumers must explicitly evaluate idempotency.
- `POST /Orders` uses an `IdempotencyKey` header generated by the client as a UUID v4.
- The HTTP-layer implementation is `IdempotentAPI` backed by Redis through `IDistributedCache`.
- The current `IdempotentAPI` package only applies to `POST` and `PATCH`; `PUT /Orders` and `DELETE /Orders` should remain semantically idempotent in the business/persistence flow unless a custom filter is introduced later.
- Do not use local memory cache for idempotency when multiple API instances are possible.
- Use distributed locks when HTTP idempotency is enabled in a horizontally scaled API.
- For critical create-order guarantees, persist idempotency state in Postgres with a unique key such as `user_id + operation + idempotency_key`.
- Repeated requests with the same key and same payload should return the original result; same key with a different payload should fail as a conflict.
- Event publication should follow the Outbox pattern: save domain state and integration event in the same local transaction, publish pending events from a worker, then mark them as published/processed.
- Event consumers should implement Inbox/deduplication by `event_id` before performing side effects.

## Authentication

- Keycloak is the local identity provider.
- Blazor Web owns user login/logout for local backoffice.
- API validates JWT bearer tokens.
- Browser-facing issuer is `localhost:8081`.
- Docker-internal authority is `keycloak:8080`.

## Authorization

- Roles/scopes are feature-oriented:
  - `products.read`
  - `products.write`
  - `orders.read`
  - `orders.write`
  - `orders.view_all`
  - `orders.view_own`
- Current `operator` can view all orders for development validation.
- Future production behavior should distinguish `orders.view_own` from `orders.view_all`.

## Frontend

- Root `/` redirects according to auth state.
- `/login` is clean and has no sidebar/topbar.
- `/logout` is a visual logged-out confirmation page.
- `/products` and `/orders` are authenticated operational pages.

## Local Infrastructure Direction

- Keep Docker Compose for current local orchestration.
- Future Kubernetes can be introduced after the local stack stabilizes.
- SOPS + Age is the preferred direction for local encrypted secrets.
- Avoid SaaS dependency for CI/CD.
